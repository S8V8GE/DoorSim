DoorSim Project Map
===================

Purpose
-------
DoorSim is a WPF desktop training/demo tool for interacting with Genetec Softwire simulated access control doors.

It is designed for local/training Softwire environments only. It is not intended for production access control systems.

DoorSim supports:
	- Manual single-door simulation
	- Manual two-door simulation
	- Cardholder drag/drop onto readers
	- PIN and Card + PIN testing
	- REX, breakglass/manual station, and door sensor interaction
	- Auto-enrolment (in manual mode)
	- Interlocking override/lockdown input controls
	- Auto Mode busy-site simulation
	- Forced-door and held-open event generation
	- Help/About windows

Models
======

AutoSimulationLogEntry.cs
-------------------------
Represents one row in the Auto Mode event log.

Used for:
	- Time text
	- Event number text
	- Log level
	- Event type
	- Door name
	- Message text
	- Separator rows between event groups
	- Colour-coding log levels such as Info, Success, Warning, and Error


Cardholder.cs
-------------
Represents a cardholder loaded from the Genetec Directory database.

Used for:
	- Displaying cardholders in the UI (manual-mode)
	- Drag/drop card presentation
	- Auto Mode cardholder selection
	- Card-only and Card + PIN simulation

Important fields include:
	- Cardholder name
	- Credential/card value
	- Trimmed credential
	- Bit count
	- PIN availability (HasPin)


CredentialFormatResult.cs
-------------------------
Represents the result of credential parsing/formatting.

Used by CredentialFormatService when converting card values into the format needed for Softwire reader simulation/auto-enrolment.


InputState.cs
-------------
Represents the state of a simulated Softwire input.

Used for:
	- Door sensor state
	- REX state
	- Breakglass/manual station state
	- Interlocking input state

Typical state includes:
	- Active/inactive
	- Shunted/not shunted


ReaderState.cs
--------------
Represents the state of a Softwire reader.

Used for:
	- Reader online/offline state
	- Reader shunted state
	- Reader LED colour/status


SimulatedInput.cs
-----------------
Represents a generic simulated input discovered from Softwire sim devices.

Used mainly by the Door Interlocking Controls area, where any available simulated input may be used for override/lockdown testing.


SoftwireDoor.cs
---------------
Represents a Softwire door and its related hardware/configuration.

This is one of the core models in DoorSim.

Used for:
	- Door display
	- Reader paths
	- REX paths
	- Door sensor path/state
	- Breakglass/manual station path/state
	- Lock state
	- Last access decision
	- Forced-door settings
	- Held-open settings
	- Card + PIN reader settings
	- Auto Mode door/candidate selection

Important note:
Softwire may display some states differently in the UI than in the API. DoorSim handles these differences where required.


Services
========

CardholderSqlService.cs
-----------------------
Loads cardholders and credentials from the Genetec Directory SQL database.

Used by MainViewModel to refresh the shared cardholder list.

The loaded cardholders are passed into CardholdersViewModel and also made available to Auto Mode.


CredentialFormatService.cs
--------------------------
Handles credential formatting/parsing logic.

Used when DoorSim needs to convert cardholder credential data into the format expected by Softwire reader simulation methods.


ISoftwireService.cs
-------------------
Interface for SoftwireService.

Used so ViewModels can depend on an abstraction rather than directly depending on the concrete SoftwireService implementation.


SoftwireService.cs
------------------
Main service for communicating with the Softwire HTTP API.

Used for:
	- Logging in / checking connection
	- Loading doors
	- Loading devices/simulated inputs
	- Reading input states
	- Setting input states
	- Reading reader states
	- Swiping raw card credentials
	- Sending Wiegand/PIN values

Important note:
This service is intended for local/training/demo Softwire environments. Certificate handling and API usage should not be treated as production security design!


SoundService.cs
---------------
Centralised service for playing DoorSim sounds.

Used for:
	- Reader credential beep
	- Access granted sound
	- Access denied sound
	- Optional/easter-egg sounds

Keeping audio here prevents sound logic being scattered across multiple ViewModels.

AutoModeViewModel.cs
--------------------
ViewModel for Auto Mode.

Responsible for:
	- Auto Mode settings
	- Event profile percentages
	- Start/stop/clear commands
	- Running counters
	- Event log entries
	- Random event selection
	- Retry guard
	- Door reservation tracking
	- Normal access events
	- Forced-door events
	- Held-open events
	- Cleanup of held-open doors

Auto Mode can generate:
	- Normal reader/cardholder events
	- Normal Card + PIN events
	- Normal REX events
	- Forced-door events
	- Held-open reader events
	- Held-open REX events

Important design:
Auto Mode does not own SoftwireService directly. MainViewModel passes it safe callbacks for the specific operations it needs.


CardholdersViewModel.cs

ViewModel for the cardholder list/panel (in manual-mode only).

Responsible for:

Responsible for:

Used by:

ViewModel for a set of doors and one selected door/panel state.

	- Single Door View

MainViewModel.cs
----------------
Main application ViewModel and coordinator.

Responsible for:
	- Softwire connection state
	- Status bar text/colour
	- Main message text
	- View mode selection
	- Manual Mode / Auto Mode switching
	- Creating and wiring child ViewModels
	- Loading doors/cardholders/interlocking inputs
	- Starting refresh timers
	- Polling selected door hardware state
	- Polling reader state

MainViewModel owns the services. Child ViewModels receive either data or narrow callbacks rather than owning service/session logic themselves.

Developer Notes
===============

Polling
-------
DoorSim uses multiple timers:
	- Slow connection/data refresh
	- Selected door hardware refresh
	- Fast reader state refresh

Reader polling is faster because LED/status changes can be brief.

Door State
----------
DoorSim keeps selected door state separate from the full refreshed door list so the UI does not flicker or lose the selected door during background refreshes.

Two Door View
-------------
Two Door View keeps left and right door panel state independent. When moving from Single Door View to Two Door View, the current single-door selection is carried into the left panel. When returning from Two Door View to Single Door View, the left panel state is carried back (the RHS panel always starts blank/empty).

Auto Mode
---------
Auto Mode generates real Softwire simulation events using callbacks supplied by MainViewModel. It deliberately avoids directly owning SoftwireService.

Held-open events reserve doors while their sensors are left open. 
This prevents another automated event from selecting the same door and accidentally interrupting the held-open scenario.

If a reserved door disappears from Softwire, DoorSim releases the reservation and continues cleanly.

Input States
------------
Softwire UI names and API state names may not always match. DoorSim generally uses the state values expected by the API when setting simulated inputs.

Shunted Inputs
--------------
When an input is shunted, Softwire may not report Active in the usual way. 
DoorSim preserves the local visual state where needed so the trainer can still understand what was done in the UI.
